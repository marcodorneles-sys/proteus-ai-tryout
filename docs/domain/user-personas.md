## How to Use This Document

Each persona entry defines who interacts with the system, what they do, why, and constraints Copilot must respect when generating specs or code for their workflows. Refer to a persona by its Persona ID (e.g., `P-01`) in feature specs and acceptance criteria.

> **Structural note (v1.1):** Following the May 29 stakeholder review with Andronic, personas are now grouped into six functional categories rather than a flat list. P-10 (Production Planner) and P-11 (PO Confirmation Operator) are no longer standalone personas; their functionality is absorbed into the relevant parent groups and gated by a factory-level SAP/non-SAP configuration flag. A new persona P-02c (Production Machine Expert) has been added. Non-production read-only consumers are consolidated into the Data Consumer group (P-04).

## Persona Structure Overview

The six functional groups below reflect Andronic's May 29 vision. Groups are ordered by operational proximity to the shop floor.

| Group | Personas | Description |
|---|---|---|
| 1. Operators | P-01 | Shop-floor users who enter production data |
| 2. Support Functions | P-02a, P-02b, P-02c | Data verification, correction, and machine expertise roles that support operators and own daily data quality |
| 3. Local Factory Admin | P-03a (Performance), P-03b (Quality) | Power users who own the system configuration at factory level — two focal points: performance (SDE/downtime) and quality (QSens) |
| 4. Data Consumers | P-04 | Read-only users across all departments; two flavors: local (factory-scoped) and global viewer (cross-factory) |
| 5. Global Admin | P-05 | Cross-factory governance, global templates, GSCA KPI consolidation (Sergey / OPI team) |
| 6. System Admin | P-06 | IT-level technical administration; exceptional access (Florin / IT team) |

> **Note:** Personas P-10 and P-11 from v1.0 are not separate personas. Production Planner functionality (P-10) belongs to the Support Functions group; it is extra functionality available to relevant support roles in non-SAP factories, not a separate person. PO Confirmation Operator (P-11) is an extension of the Operator role in non-SAP factories. Both are gated by factory-level SAP/non-SAP configuration and must not be provisioned as distinct roles.

## Group 1 — Operators

The largest user group. Blue-collar workers stationed at or near a production line whose primary job is running the line. The system is a side tool, not their core work.

---

### P-01 — Shop-floor Operator (Equipment Owner)

| Attribute | Value |
|---|---|
| **Role code** | `biz.SFOperator` |
| **Also known as** | Line crew, Equipment owner, Machine operator, Technician, Electrician, Mechanic, Automation |
| **Organizational scope** | Factory / production line level |
| **Factory access** | Single plant, single or paired production line |
| **Non-SAP variant** | In non-SAP factories: includes PO confirmation capability (no separate role required — gated by factory configuration flag) |

**Description**

The largest user group in the system. Blue-collar workers stationed at or near a production line. Their primary job is running the line and producing product — the system is a side tool, not their main job. Users rotate across shifts (day/night/3rd) and are not digitally elevated.

Physical context (confirmed May 29): the shop-floor terminal (typically a 17-inch screen on a pole or mounted under a conveyor belt) may be 5–30 metres from the machine. The operator must walk to it, log in on a generic shared account, complete their entry, and return to the line. This physical separation is a primary driver of data quality issues and must be reflected in UX design.

Access is via a shared, dedicated terminal on or near the production floor (kiosk-style), where any qualified line crew member can log in with a shared shop-floor credential without individual login.

**Goals**

- Record machine downtime reasons and reclassify automated MDC stops as quickly as possible without breaking production flow
- Enter reject quantities and map them to reject reason codes per production order
- Check sensor readings via QSens Cockpit and confirm checks with minimal clicks
- Get in and out of the system fast — every extra step is friction

**Key Tasks**

- Enter or reclassify **downtime** stops in Smart Downtime Entry (SDE)
- Perform bulk reclassification of multiple stops of the same type
- Enter actual **machine speed** per shift; a reason code is **mandatory** (lock rate / deviation — e.g., machine condition, spare part change); the system must not allow saving speed entry without a reason
- Submit **reject quantities** and link them to production orders
- Perform sensor checks in QSens Cockpit and record off-reason explanations for out-of-range parameters
- View machine performance (Office Screen)
- View current production order context, **PO volumes** (Information Header in SDE)
- Perform **changeovers**
- Non-SAP factories only: add, edit, and delete PO confirmations (enabled by factory configuration flag, not a separate provisioned role)
- Data modifications are allowed only within the shift window and within the shift handover window (configurable per factory; default 30 minutes after shift end — confirmed May 27)

**Permissions / Access Scope**

- Write access: current shift downtimes only; previous shift data editable only within shift handover window
- No access to master data configuration forms
- No access to cross-factory data
- No access to KPI reports or operational reports beyond shift-level dashboards visible in SDE (except for performance reports in SDE which includes data from up to 5 shifts ('Statistics' tab) to compare their shift performance with the performance numbers of other shifts)
- Cannot delete MDC-generated stop records — can only reclassify them

**Frequency & Context of Use**

- Multiple times per shift, every shift (24/7 rotating schedule)
- Used at a dedicated kiosk terminal on the factory floor, often in noisy, physically demanding conditions
- User may have greasy hands or be interrupted frequently; interaction must be fast and fault-tolerant

**Pain Points (Previous system version)**

- Drop-down menus with 20+ options are difficult to scroll through on a shop-floor terminal
- Entering start time, end time, and duration manually was error-prone and slow
- No visual timeline of machine stops — had to enter all downtime data without context
- Long, complex forms with many required fields caused operators to delay or skip entries
- System slowness directly delayed operators' ability to leave at end of shift
- Current system: No self-service correction — operators cannot review their own speed or reject entries within SDE — only managers/analysts can do that (confirmed known gap and planned improvement, May 27)

> **Spec Hints for Copilot**
> - All UI flows for this persona must follow wizard/step-by-step patterns: one decision at a time, large tap targets, minimal cognitive load — not form-based layouts; treat SDE as a kiosk / shop-floor industrial environment application, not a standard office web form
> - The SDE UI is the most critical interaction surface for this persona
> - Never generate specs that require this persona to manually enter timestamps — the system must derive them from MDC data or provide pre-filled suggestions, except for cases when there is no data coming from machine
> - Shift handover locking is a hard constraint: Copilot-generated specs must enforce that write access to previous-shift data expires after the configured handover window (default 30 min)
> - Shared login (shop-floor user credential) means no user-specific preferences or personalization should be assumed for this persona
> - Reclassification business rules (Override Runtime, Ignore MDC Stop, Min. duration, Can insert into Future, Long lasting) are embedded in the system — Copilot must never expose these as user-facing choices; they filter available options automatically
> - Speed entry must enforce a mandatory reason code; do not generate specs that allow saving speed data without a reason
> - Self-correction for speed and reject entries is a planned improvement — do not generate specs that assume this capability exists in the initial release
> - PO confirmation for non-SAP factories must be gated by a factory-level configuration flag, not provisioned as a separate security role

## Group 2 — Support Functions

Roles that sit close to the shop floor and support operators: verifying data correctness, resolving discrepancies, and owning machine expertise. In smaller factories, one person may cover all three sub-roles. In larger factories the roles are distinct. The system must not enforce a fixed role-to-person binding.

---

### P-02a — Process Lead / Shift Supervisor (Data Verification)

| Attribute | Value |
|---|---|
| **Role code** | `biz.ShiftLeader` (closest mapping; IWS-specific title varies by factory) |
| **Also known as** | Shift leader, Group leader, Team lead, Cell lead, SAP operator |
| **Organizational scope** | Factory / cell or production area level |
| **Factory access** | Single plant; access to multiple production lines within their area |
| **Non-SAP variant** | May include production order creation capability (gated by factory configuration, not a separate role) |

**Description**

A senior shop-floor or office role responsible for verifying, correcting, and owning production performance data for their shift or area. Under the IWS (Integrated Work Systems) methodology now being rolled out, this persona is the IWS Cell Lead's system counterpart.

Critical design constraint: the mapping between job titles and tasks varies significantly by factory. A small factory like Ukraine may have a single person doing everything described across P-02a and P-02b. A larger factory may have a team of 12 sharing the same work. The system must not enforce a specific role-to-task binding.

**Goals**

- Primary goal: responsible for data accuracy and completeness of all data entities used in KPI calculations and reporting
- Ensure data is reviewed and corrected in time (timeliness) and stays accurate
- Known pain point: if data is not verified in a timely manner, downstream consumers see incorrect/misleading numbers for prolonged periods and observe significant KPI swings after corrections
- Verify that all shift downtime data is correctly classified before the shift closes
- Check production volume figures against SAP confirmations and resolve minor discrepancies
- Ensure OEE and MTBF figures are accurate for daily direction-setting meetings
- Review and adjust shift KPI data so it can flow downstream

**Key Tasks — Daily Data Verification Sequence (order matters)**

1. Review downtime entries in SDE: check for MDC data gaps, unclassified stops >10 minutes, production sequence correctness
2. Check actual speed entries: confirm all machines have speed recorded; flag entries not at lock rate without a valid reason
3. Review Import Issue Resolution report: confirm no unprocessed SAP confirmation errors
4. Review PO Confirmation Resolution report: confirm no confirmations mismatched to the wrong shift or PO
5. Review Shift Remainders: investigate any reminder exceeding one pallet
6. Review Reject Entry: check for production periods without reject entry, and entries without associated production
7. Review OEE / KPI reports: confirm MTBF, unplanned stop count, attainment to plan, maker reject ratio, process failures, breakdown count, changeover time variance

**Key Tasks — Other**

- Monitor Office Screen for live shift performance across all lines
- Participate in or prepare data for Daily Direction Setting (DDS) meeting
- Enter missing downtimes that operators failed to log during the shift
- Non-SAP factories only: create production orders in GSCA Machinery when SAP is not the source (gated by factory configuration)

**Permissions / Access Scope**

- Write access: current and previous shifts (broader window than Operator)
- Read access to OEE reports and downtime statistics
- Cannot modify master data configuration
- Limited to their assigned plant's data

**Frequency & Context of Use**

- At the end of each shift and at the start of the following day
- Office or control room environment — standard desktop terminal
- Cell leads use Office Screen on ~12-inch tablets during DDS meetings
- Time-pressured: needs to complete review before the next shift and before daily direction-setting meetings

**Pain Points (Legacy MII)**

- No clear audit trail of who entered or changed which downtime record
- Reconciling SAP volume data with GSCA Machinery data required manual comparison across two systems
- No consolidated view of all lines' performance — had to navigate per-machine or per-workcenter
- Delayed data from SAP (up to 30-minute lag) sometimes caused review to be based on incomplete figures

> **Spec Hints for Copilot**
> - This persona uses both SDE (for corrections) and operational reports (for review) — do not design these as completely separate flows; the transition between them should be seamless
> - PO Confirmation Resolution and Import Issue Resolution are daily tasks for this persona, not exception screens — design for efficient zero-result confirmation (green = done, no action needed)
> - Specs must support bulk operations (bulk reclassification, bulk resolution) — single-record workflows are insufficient at line-level volume
> - The shift handover window must be wider for this role than for P-01; the exact duration is factory-configurable
> - Avoid generating specs that assume a single shift schedule model — factory schedules vary and month-end schedules have special 3-segment structures
> - A unified data health dashboard is a confirmed planned improvement — generate specs that anticipate this as a landing point for this persona's daily workflow
> - The UI/UX for this persona is office environment: broader, multi-line views appropriate for desktop — do not apply shop-floor kiosk patterns (see P-01)
> - Responsive layout for tablet viewports (~1024px width) must be considered for Production Manager / Cell Lead use
> - Production order creation for non-SAP factories must be gated by a factory-level configuration flag, not provisioned as a separate security role
> - Daily data locking (marking a shift's data as human-verified) is a confirmed planned future enhancement — design verification workflows to leave a clear integration point, but do not include in initial release specification

### P-02b — Production Analyst (KPI & Period Management)

| Attribute | Value |
|---|---|
| **Role code** | `biz.ProdOffice` / GSCA Global Admin role (varies) |
| **Also known as** | Production Process Manager, Process Analysis & Improvement Manager, Automatic Systems Engineer |
| **Organizational scope** | Factory level (local) or cross-factory (global variant) |
| **Factory access** | Single plant (local) or multiple plants (global variant) |

**Description**

A technical-analytical office role responsible for the integrity and accuracy of production KPI data at the factory level. Retains primary ownership of month-end KPI locking, SAP reconciliation, PO confirmation resolution for complex cases, and factory-level reporting. In some factories, this persona also covers daily data verification tasks described under P-02a — the boundary is factory-specific and fluid.

Important: there is no system-enforced data release gate — downstream consumers see the current data state at all times. The verification process exists to ensure data is correct before the KPI period locks, not to control access.

Scope note: local-factory Excel templates (e.g., Ukraine's production Excel reports) are explicitly out of scope for the new platform. The raw data export API is being deprecated.

**Goals**

- Ensure all production KPI data is clean, reconciled with SAP, and locked before the KPI period closes (typically day 5–7 of the following month)
- Identify and resolve data anomalies (PO confirmation mismatches, unrecorded downtimes, SAP sync gaps)
- Generate and validate monthly KPI reports for distribution to factory management and GSCA Global
- Maintain data quality standards so downstream consumers (SAP BW, Data Lake, Power BI) receive accurate data

**Key Tasks**

- Daily: review PO confirmation resolution screen; resolve delayed or misaligned SAP confirmations
- Daily: check non-recorded (unaccounted time) metrics across all lines for anomalies
- Monthly: verify all production orders are reconciled between GSCA Machinery and SAP (zero discrepancy target)
- Monthly: lock the KPI period (triggered automatically on day 5–7; analyst confirms data is clean first)
- Monthly: review reject entries per line for completeness and accuracy
- Review operational reports: OEE, availability, MTBF, downtime analysis, reject ratio
- Manage and validate shift schedules align with production calendar (including month-end 3-segment schedule)
- Escalate data quality issues to Process Leads or SAP teams for correction

**Permissions / Access Scope**

- Read access: all operational reports and KPI reports for their plant
- Write access: downtime and reject records within the open KPI period (broader than Process Lead)
- Write access: PO confirmation resolution
- Write access: KPI corrections (within period)
- Read access to system audit log (who changed what, when)
- Write access: shift schedule
- Cannot modify system configuration or master data structure

**Frequency & Context of Use**

- Daily: 30–60 minutes of data review and reconciliation
- Month-end: intensive 1–2 day effort to close the period cleanly (cascading impact on Finance, GSCA Global, SAP BW)
- Office environment, standard desktop; also accesses Power BI dashboards fed from Data Lake
- Works across multiple system views in a single session (SDE → OEE report → PO resolution → schedule)

**Pain Points (Legacy MII)**

- No single consolidated daily view across all lines — had to navigate machine by machine
- Month-end closure historically took 4–15 days of corrections (now reduced but still manually intensive)
- Raw data export API was the only way to build custom local reports — now being deprecated
- Data Lake Power BI reports update with a 4-hour lag, making them useless for same-day month-end decisions
- No favourites / quick-navigation system that persists across screens meaningfully

> **Spec Hints for Copilot**
> - The favourites / bookmarked reports feature is explicitly valued by this persona (confirmed by Stanislav); should be included in the new system
> - Month-end KPI locking is a time-critical, high-stakes workflow — specs must include explicit confirmation steps, period status indicators, and guard rails preventing data modification after lock
> - The non-recorded time indicator is a key data quality signal; it must be visible in the daily OEE report without requiring a separate query
> - PO confirmation resolution must support bulk resolution (system-suggested allocation) as well as manual per-record resolution
> - Do not generate specs that expose raw SQL query endpoints or direct database connections unless explicitly requested (not planned for the initial implementation)
> - Cross-shift and cross-period data views are essential — single-shift-only views are insufficient
> - Do not generate specs for a data release gate or verified status that blocks other users from reading data — KPI period locking is the only data-freeze mechanism
> - Do not generate specs that replicate local-factory Excel report patterns — those are explicitly out of scope

### P-02c — Production Machine Expert *(New in v1.1)*

| Attribute | Value |
|---|---|
| **Role code** | TBD — to be defined |
| **Also known as** | Machine data expert, Automation engineer, MDC mapping specialist |
| **Organizational scope** | Factory level |
| **Factory access** | Single plant — deep technical knowledge of machines in that plant |
| **Source** | Added per Andronic's May 29 review session; not present in v1.0 |

**Description**

A technically specialized role focused on the inner workings of production machines and their data representation in the system. This persona owns the mapping between physical machine events (stops, rejects) and the system's classification structures (breakdown catalogs, reject catalogs), and also owns the configuration of machine sensor parameters in QSens.

In some factories, this role is combined with the QSens Local Admin role (P-03b). In others, the machine expert provides technical knowledge (which sensors/parameters to use) while a separate quality specialist performs the configuration. The system must support both patterns.

This persona is typically a technician or automation engineer. They cannot be substituted by a process lead or production analyst because the role requires deep machine engineering knowledge.

**Goals**

- Own and maintain the correct mapping between physical machine breakdown events and the system's stop/reject classification structures
- Ensure QSens sensor tag codes are correctly identified and mapped to machine parameters
- Provide technical expertise to QSens Local Admins who configure business parameters but lack machine engineering knowledge
- Serve as the authoritative source for machine behaviour and sensor semantics within the factory

**Key Tasks**

- Define and maintain MDC breakdown-to-reason mappings (global library → local library)
- Define and maintain reject cause mappings for machine-level events
- Define and maintain MDC breakdown catalogs
- Identify correct PCO tag codes for QSens sensor parameters
- Validate that configured sensor parameters reflect actual machine measurement points
- Advise QSens Local Admins on sensor configuration questions requiring machine engineering knowledge
- Support factory go-live: verify all machines' MDC mappings are complete and correct before activation

**Permissions / Access Scope**

- Write access: MDC breakdown and reject mapping forms
- Write access: QSens sensor tag code configuration (technical parameter mapping, not business threshold configuration)
- Read access: all reports related to machine performance and sensor utilization
- No access to shift schedule, user management, or financial configuration
- May overlap with `biz.QSensConfigAdmin` in factories where the machine expert doubles as QSens Local Admin

**Frequency & Context of Use**

- Project-phase intensive during factory setup and MDC integration
- Ongoing: triggered by machine changes, new equipment additions, or breakdown classification updates
- Office or control room environment; also physically visits the production floor to verify machine hardware

> **Spec Hints for Copilot**
> - This persona is the technical counterpart to QSens Local Admin (P-03b) — machine expert owns what to measure (tag codes, physical mapping); QSens Local Admin owns how to evaluate it (business thresholds, adherence rules)
> - MDC mapping forms must be accessible to this persona without requiring Local Config Admin (P-03a) privileges — mapping is a specialist technical task, not a general admin task
> - Specs for breakdown and reject mapping forms must support a two-level structure: global library (managed by Global Admin P-05) and local library (managed here) — this persona works at the local level and cannot modify global definitions
> - When this persona is combined with QSens Local Admin in a single factory user, the permissions of both personas apply without requiring two separate accounts

## Group 3 — Local Factory Admin

Power users who own the system at factory level. Andronic confirmed in the May 29 session that there are two focal points for this group: a performance-side admin (owns SDE, downtime configuration, production setup) and a quality-side admin (owns QSens). In some factories, these are the same person; in others, they are distinct individuals. The RBAC design must support both patterns.

### P-03a — Local Configuration Administrator (Performance Side)

| Attribute | Value |
|---|---|
| **Role code** | `biz.ConfigAdmin` |
| **Also known as** | Power user, Local GSCA Power User |
| **Organizational scope** | Factory level |
| **Factory access** | Single plant — full configuration authority for that plant |

**Description**

The local system owner at factory level for the performance/production side of GSCA Machinery. Responsible for configuring the platform to match the factory's current operational setup: machines, workcenters, shift schedules, downtime reason catalogs, nominal speeds, and user access. Acts as the first line of support for factory users and as the liaison to GSCA Global for configuration questions. Can play the P-02c — Production Machine Expert role in some factories.

Accountability: this persona is responsible for all permissions they create. They need a clear view of who holds which roles in their plant to fulfil this accountability.

**Goals**

- Keep system configuration aligned with actual factory operations
- Provision and manage factory user access: responsible for permission requests and approvals in MyIT portal
- Maintain accountability for all permissions created — requires a role-assignment overview screen
- Maintain downtime classification structures (locations, reasons, machine-to-reason mappings)
- Support local users when they encounter data or system issues

**Key Tasks**

- Create and maintain the plant/workcenter/machine hierarchy (~20 forms)
- Manage shift schedule templates including month-end special schedules
- Maintain downtime location and reason catalogs (~12 forms), including machine-to-reason mapping matrices
- Manage user accounts (including generic accounts, like those used by machine operators or to display dashboards on the big screen on a factory): create, assign roles, assign plant access, set their standard language
- Maintain local translation strings (local language for system labels and object names)
- Configure workcenter groups for reporting filters
- Request for the nominal speed (rate) to be set by the P-05 Global Admin
- Set brand change time for workcenters (initial setup only; may be locked thereafter)
- Configure notification recipient lists for automated mail reports
- Review data entered by operators and perform corrections beyond the period allowed to Process Leads

**Permissions / Access Scope**

- Full configuration access for their assigned plant
- Cannot modify global templates — read only
- Cannot modify business data for closed KPI periods
- Can grant/revoke application-level roles for users in their plant
- Cannot access other factories' configuration or data

**Frequency & Context of Use**

- Configuration tasks: as needed, not daily — triggered by factory changes, new hires, or process updates
- Data support: several times per week
- Office environment, standard desktop
- During factory go-live or major changes: intensive usage for 1–2 weeks

**Pain Points (Legacy MII)**

- User provisioning required two manual steps: SAP platform account creation (external) + application-level role assignment — not integrated
- To translate a system object for a local language, had to switch the UI language first — no single multi-language edit form
- No automated sync between SAP master data and GSCA Machinery — changes in SAP required a second manual update in GSCA
- Adding a new production line required raising a ticket to BTS IT — not self-serviceable

> **Spec Hints for Copilot**
> - The ~90 admin forms are largely templated CRUD — generate consistent list + detail form patterns; do not generate unique layouts per form unless functional requirements explicitly differ
> - Translation management must allow editing local-language values without switching the UI language
> - User provisioning specs must account for a two-layer model: Azure Entra ID account (managed externally) + application role assignment (managed here). Do not conflate these.
> - Dynamic workcenter management (adding/removing workcenters without redeployment) is a planned enhancement — do not assume a static list
> - Shift schedule forms must support the month-end 3-segment pattern (pre-midnight / post-midnight / next day) as a first-class configuration option
> - SAP master data is already being synced into the GSCA Machinery database but is not yet surfaced in admin UI forms. Design for SAP-driven pre-population with power user confirmation, not purely manual creation.
> - This persona must have a role-assignment overview screen: who holds which role in their plant — required to fulfil their accountability for granted permissions

### P-03b — QSens Local Admin (Quality Side)

| Attribute | Value |
|---|---|
| **Role code** | `biz.QSensConfigAdmin` |
| **Also known as** | Local quality specialist, QSens configuration owner |
| **Organizational scope** | Factory level — QSens module specifically |
| **Factory access** | Single plant, QSens configuration and monitoring |
| **Note** | In some factories, combined with P-03a (Local Config Admin) or P-02c (Production Machine Expert). System must support all patterns without separate accounts. |

**Description**

A quality department representative responsible for configuring QSens sensor definitions at factory level using globally defined templates as a baseline, monitoring sensor adherence, and maintaining local quality parameter settings. Distinct from the Shop-floor Operator who performs sensor checks — this persona defines the rules those checks are validated against.

Boundary with P-02c (Production Machine Expert): the machine expert identifies correct technical tag codes and physical mapping; the QSens Local Admin configures business parameters (thresholds, adherence rules, formula types) against those mapped sensors.

**Goals**

- Ensure QSens sensor templates are correctly mapped to local machine parameters
- Configure adherence rules and thresholds that accurately reflect quality requirements
- Monitor sensor adherence KPIs to identify quality trends

**Key Tasks**

- Map global sensor template parameters to local machine parameter tag codes
- Configure business parameters: target values, adherence thresholds, formula type per sensor
- Set local-language labels for sensor names displayed in QSens Cockpit
- Review sensor utilization and adherence KPI reports
- Coordinate with P-02c on correct tag code identification

**Permissions / Access Scope**

- Write access to QSens sensor configuration for their plant
- Read access to all QSens reports
- No access to SDE configuration
- Cannot modify global QSens templates — read only

**Frequency & Context of Use**

- Configuration: project-phase intensive, then as-needed for parameter changes
- Monitoring: daily or weekly review of adherence reports
- Office environment, standard desktop

**Pain Points (Legacy MII)**

- Global parameter template synchronization was done via circulated Excel files — error-prone and slow
- No way to validate configured parameters match the Excel template without manual line-by-line comparison
- Sensor formula types (8 variants) are complex to configure without tooling support

> **Spec Hints for Copilot**
> - QSens formula types (True/False, Greater/Lower, Range-Based, Control Degree Calculated, Control Limit Calculated, Percentage Control Limit, Range Ruled Control Limit, Range Ruled SD Control Limit) are enumerated business logic — represent as constrained selection, not free-text
> - The QSens Cockpit is bidirectional (read + write) — sensor check confirmation is a write operation; never generate this screen as read-only
> - Global template to local configuration sync is a known pain point and planned enhancement; leave a clear integration point in configuration form specs
> - When combined with P-02c (Production Machine Expert), ensure machine-side tag code configuration and quality-side business parameter configuration are distinguishable in the UI
> - The Office Screen also serves a secondary use case: MDC fit testing — verifying a newly connected machine's data is flowing correctly. Ensure refresh cycle is fast enough to support this.

## Group 4 — Data Consumers (Read-Only)

Everyone who consumes system data without modifying it. Andronic confirmed in the May 29 session that A1 does not operate a report-level security model inside GSCA Machinery — confidential production processes are not run through the system, so whatever data is in the system can be seen by any authorized viewer. There is no need to split report access by department.

---

### P-04 — Data Consumer

| Attribute | Value |
|---|---|
| **Role code** | `biz.ReadOnly` (local) / `biz.GlobalViewer` (global) |
| **Also known as** | Local viewer, Global viewer, Finance user, Quality observer, Production Manager, Production Director, Non-production office user, Make & Pack Manager, Cell structure lead |
| **Organizational scope** | Factory level (local) or cross-factory (global) |
| **Factory access** | Local: single plant. Global: all plants (auto-provisioned via AD group). |

**Local Read-Only**

Users within a factory who need visibility into system data but have no right to modify anything. Includes production managers, directors, finance representatives, quality observers, visiting engineers, and users being onboarded. Provisioned locally by the Local Config Admin (P-03a).

**Global Viewer**

Users who need cross-factory read access regardless of which plants exist. Provisioned via a single global AD group so that when a new factory is added, no per-factory provisioning is needed. Designed for OPI team members, global governance roles, and other central functions who need to see data across all factories.

Pain point addressed: in the current system, global users must be manually added to each factory's AD group. When a new factory is created, someone must remember to add all global viewers. The global viewer group eliminates this by using a single overarching AD group — membership is managed by the group owner, not per-factory.

**Goals**

- Access production data relevant to their function (reports, dashboards, Office Screen) without impacting data integrity
- Run specific operational reports without navigating the full system
- Monitor real-time production performance (Production Manager use case)

**Key Tasks**

- Run operational and KPI reports
- Monitor the Office Screen (live shift KPI overview)
- View downtime statistics and OEE trends
- Access DDS-relevant KPIs for daily direction-setting meetings (Production Manager / Cell Lead use case)

**Permissions / Access Scope**

- Read-only access to all reports and dashboards
- No write access of any kind
- No configuration access
- Local flavor: scoped to a single plant
- Global flavor: scoped to all plants

**Frequency & Context of Use**

- Production Manager / Director: Office Screen continuously during working hours (often on a dedicated large display); KPI reports daily and monthly
- Finance / Quality / other departments: periodic, often at month-end or on-demand
- Global Viewer: remote access across time zones

> **Spec Hints for Copilot**
> - Color coding on the Office Screen (green/red/blue/gold/grey/orange workcenter states) is a hard functional requirement, not a cosmetic choice
> - Office Screen must support drill-down to a per-machine KPI popup showing at minimum: reject ratio, top stops, volume attainment, and MTBF — without navigating away from the main screen
> - Do not generate specs that require this persona to configure anything — pure read-only consumer
> - No report-level permission configuration is needed — A1 does not operate a report-access security model; do not generate specs that add per-user report filtering
> - All write controls must be hidden or disabled for this role — prefer hiding over showing disabled controls
> - API endpoints must enforce read-only at the authorization layer, not only in the UI
> - Global Viewer provisioning must use a single AD group — do not generate specs that require per-factory provisioning for global viewers

**Description**

The global business owner of the GSCA Machinery platform. Responsible for cross-factory data quality, global KPI consolidation, global master data stewardship, and platform governance. Acts as the demand control gate for change requests from factories. Manages global configuration objects (global machine sensor templates, KPI period definitions, global downtime catalogs, global breakdown/reject library) that local admins can reference but not modify.

**Goals**

- Ensure consistent, comparable KPI data across all GSCA factories globally
- Govern change requests and feature demands from factory power users
- Maintain global reference data (templates, catalogs, global library) used by all factories
- Trigger and validate the GSCA KPI consolidation and distribution to SAP BW

**Key Tasks**

- Review and validate SAP EAM (plant hierarchy) data downloaded to GSCA Machinery
- Manually input Shift Schedule, Targets, Volume and other global data when required
- Manage global QSens sensor templates and business parameter definitions
- Maintain global library of downtime reasons and breakdown/reject classifications
- Trigger GSCA KPI data consolidation and distribution to SAP BW
- Notify local engineering departments of missing or incorrect SAP data
- Govern user demand: filter and prioritize change requests from factories before forwarding to IT
- Set nominal speed (rate) per the request from the P-03a Local Admin

**Permissions / Access Scope**

- Read and write access across all plants
- Write access to global configuration objects (templates, global catalogs, global library)

- Monthly: GSCA KPI consolidation and SAP BW distribution

> - This persona is the sole owner of the global breakdown/reject classification library — local mapping forms must clearly show global definitions are read-only at the local level
> - This persona is not a frequent direct user of SDE or shop-floor screens — do not default to shop-floor UX patterns for their workflows

**Description**

An IT-level technical administration role for exceptional, low-frequency tasks that require direct system access beyond what the business configuration UI provides. Examples include changing time zones, parameterising hard-coded values, managing edge server connections, and resolving database-level issues.

Andronic's framing (May 29): *"System admin is for IT — people like Florin who need to change a time zone or something critical that needs a bit of technical understanding of the system. Instead of hard-coding some stuff, maybe we parameterise and have access to this."*

This persona does not run the system on a daily basis and is not a primary actor in business workflows.

**Goals**

- Maintain platform-level technical configuration that cannot be managed through the standard admin UI
- Resolve infrastructure and integration issues escalated from factory or global admins
- Parameterise system-level settings to reduce hard-coded dependencies

**Permissions / Access Scope**

- Full platform access including technical configuration forms not accessible to business users
- Database-level access for exceptional corrections (intended to be reduced via parameterisation in the new system)
- No business process ownership — actions are technical, not operational

> **Spec Hints for Copilot**
> - This persona should be used as infrequently as possible — system design should aim to surface parameterisable settings to Global Admin (P-05) or Local Config Admin (P-03a) rather than requiring IT intervention
> - Copilot should not generate functional requirements for this persona's workflows — their actions are outside the standard application spec scope
> - Any functionality that currently requires System Admin intervention should be flagged as a candidate for a dedicated admin UI feature in the new system
